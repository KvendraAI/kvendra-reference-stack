# Signing & verification

The Kvendra container images are signed using **Sigstore/cosign keyless**
(GitHub Actions OIDC) — no PKI keys to manage, no certificate authorities
to trust beyond Sigstore's public infrastructure (Fulcio + Rekor).

This document explains:

1. What is signed and how to verify it.
2. The trust chain (GitHub Actions identity → signed image).
3. How to verify SBOM attestations.
4. How to (optionally) verify upstream images (Postgres / Ollama).

## What is signed

For each Kvendra-produced image (currently `kvendra-platform`), the
release pipeline produces:

- **Image signature** — proves the image was built and pushed by the
  GitHub Actions workflow in `KvendraAI/kvendra-platform`.
- **SBOM attestation** (SPDX JSON) — a signed Software Bill of Materials
  attached to the image digest.

Both live in Sigstore's transparency log (Rekor) and reference the
GitHub Actions OIDC identity used at build time.

## Quick verify

```bash
./scripts/verify.sh
```

This is the canonical entry point for the reference stack. It runs the
cosign verifications below for each pinned Kvendra image. It fails if any
signature does not check out — you can override with `SKIP_VERIFY=1`
but you really shouldn't.

## Manual verification

```bash
IMG=ghcr.io/kvendraai/kvendra-platform:0.1.0-alpha.0

# 1. Signature
cosign verify "$IMG" \
  --certificate-identity-regexp '^https://github\.com/KvendraAI/' \
  --certificate-oidc-issuer https://token.actions.githubusercontent.com

# 2. SBOM attestation
cosign verify-attestation "$IMG" \
  --certificate-identity-regexp '^https://github\.com/KvendraAI/' \
  --certificate-oidc-issuer https://token.actions.githubusercontent.com \
  --type spdxjson
```

The `--certificate-identity-regexp` pins the identity to anything under
the `KvendraAI` GitHub org. Tighten further if you want to require a
specific repository / workflow:

```bash
--certificate-identity 'https://github.com/KvendraAI/kvendra-platform/.github/workflows/release.yml@refs/tags/v0.1.0-alpha.0'
```

The `--certificate-oidc-issuer` is GitHub's OIDC issuer. Other valid
issuers (GitLab, BuildKite, Google Cloud Build) would have different
URLs; we only use GitHub Actions today.

## Trust chain

```
   ┌────────────────────────────────────┐
   │ GitHub Actions OIDC token          │
   │   sub: repo:KvendraAI/...:ref:tag  │
   │   iss: token.actions.github.com    │
   └─────────────────┬──────────────────┘
                     │
                     ▼
   ┌────────────────────────────────────┐
   │ Sigstore Fulcio                    │
   │   issues short-lived cert tying    │
   │   the OIDC identity to a key       │
   └─────────────────┬──────────────────┘
                     │
                     ▼
   ┌────────────────────────────────────┐
   │ cosign sign                        │
   │   signs image digest with the      │
   │   short-lived key, uploads to      │
   │   Sigstore Rekor (transparency)    │
   └─────────────────┬──────────────────┘
                     │
                     ▼
   ┌────────────────────────────────────┐
   │ Image @ ghcr.io with signature     │
   └────────────────────────────────────┘
```

Verification is the reverse: cosign fetches the signature, retrieves the
matching Rekor entry, validates the certificate chain back to Sigstore's
public CA, and confirms the OIDC identity matches the
`--certificate-identity-regexp` you provided.

## SBOM

The SBOM is a SPDX-JSON document listing every package, library, and
binary inside the image. It is generated by [Syft](https://github.com/anchore/syft)
during the release pipeline and **attested** (signed) by cosign in the
same step.

To extract the SBOM for a published image:

```bash
cosign download attestation \
  ghcr.io/kvendraai/kvendra-platform:0.1.0-alpha.0 \
  | jq -r '.payload' | base64 -d | jq '.predicate' > sbom.spdx.json
```

The SBOM is also attached as an asset on the GitHub Release page —
that's the easier way to grab it for archives or compliance reports.

## Verifying upstream images

The stack also runs:

- `pgvector/pgvector:pg16` — Docker Hub library image, signed via Docker
  Content Trust if you opt in.
- `ollama/ollama:0.24.0` — Ollama's official image.
- `postgres:16-alpine` — official Postgres image.

We do not sign these (they are not Kvendra products). If your security
policy requires verifying them:

- **Docker Content Trust** (legacy Notary): `export DOCKER_CONTENT_TRUST=1`.
- **Pinning by digest**: edit `docker-compose.yml` to use
  `image: pgvector/pgvector@sha256:<digest>` instead of the tag.

Or rebuild from source — see `docs/troubleshooting.md` § "Fully
from-source build".

## Release pipeline (for maintainers)

The release pipeline lives at
[`.github/workflows/release.yml`](https://github.com/KvendraAI/kvendra-platform/blob/main/.github/workflows/release.yml)
in the `KvendraAI/kvendra-platform` repo. It triggers on `v*.*.*` tag
push and:

1. Builds a multi-arch image (`linux/amd64,linux/arm64`).
2. Pushes to `ghcr.io/kvendraai/kvendra-platform:<version>` + `:latest`.
3. Signs the digest with cosign keyless (GHA OIDC).
4. Generates the SPDX SBOM with Syft.
5. Attests the SBOM to the digest with cosign.
6. Creates a GitHub Release with the SBOM as an asset.

No secrets are needed beyond the default `GITHUB_TOKEN` — the
`id-token: write` permission gives the workflow access to the OIDC
token that Sigstore Fulcio exchanges for a short-lived signing
certificate. **There are no long-lived keys to rotate or leak.**

## Frequently asked

**Q: Why keyless instead of a project-controlled key?**
Keyless avoids a long-lived secret in the org. Compromising the key is
no longer possible because there is no key — every signature uses a
fresh certificate that expires in 15 minutes. The trust is in the OIDC
identity (GitHub Actions in the `KvendraAI` org) and Sigstore's
transparency log.

**Q: What if Sigstore Rekor is down?**
Verification fails until it is back. For air-gapped environments, see
[`cosign verify --insecure-ignore-tlog`](https://docs.sigstore.dev/cosign/working_with_blobs/#verify-without-rekor)
(not recommended for normal use).

**Q: Where do I report a key compromise?**
There is no key to compromise. If a GitHub Actions workflow is
modified to sign malicious content, that would be visible in the
Rekor log (the OIDC identity would be the same, but the artifact
hash would change). Report security issues via the channel in
[`SECURITY.md`](https://github.com/KvendraAI/kvendra-platform/blob/main/SECURITY.md)
of the affected repo.
